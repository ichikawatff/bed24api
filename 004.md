# Beds24 API連携 自社閲覧アプリ仕様書（第4案：最適化バイブコーディング版）

これまでの3つの案（001.md, 002.md, 003.md）の長所と短所を分析し、AIを活用した「バイブコーディング」で最も効率的に、かつBeds24 APIの厳しい制限をクリアしながら実装できる「最適解（第4案）」を提案します。

## 1. 各案の長所・短所（分析）

| 案 | 特徴 | 長所 | 短所 |
| :--- | :--- | :--- | :--- |
| **001.md** | 標準的・網羅的なWeb設計 | 全体像が詳細で、テストやセキュリティ要件が網羅されている。 | フロント/バック/DBが分離しすぎており、AI（バイブコーディング）の指示出しが煩雑になりやすい。 |
| **002.md** | Beds24の仕様に特化した堅牢設計 | APIクレジット制限やWebhookの重要性を考慮。閲覧専用にするなど要件の切り出しが現実的。 | DB設計が詳細すぎるため初手からAIに作らせるには重く、エラー解決の工数がかさむ可能性がある。 |
| **003.md** | バイブコーディング・売上可視化特化 | Next.js+SupabaseのAIが得意なスタック。日割り売上のDB構造の定義が経営視点として秀逸。 | 在庫同期やWebhookなどBeds24特有のAPI仕様への言及が欠けており、運用時にレートリミットで詰む危険がある。 |

## 2. 第4案（最適解）のコンセプト

**「003のAI特化スタックと売上管理」＋「002のBeds24 API仕様適応（Webhookと閲覧専用）」のハイブリッド**

AIがエラーなく爆速でコードを生成でき（Next.js + Supabase）、かつBeds24のAPIクレジット制限に抵触しない（Webhook + 差分同期）、さらに経営に必要な「日割り売上グラフ」と「カレンダー在庫状況」を両立させる構成です。

---

## 3. 技術スタック（バイブコーディング最適化）

- **フロントエンド ＆ APIサーバー**: `Next.js (App Router, TypeScript)`
  - 理由: APIエンドポイントと画面を同一リポジトリで管理できるため、AIに全体像のコンテキストを渡しやすく、破綻しにくい。
- **データベース ＆ 認証**: `Supabase (PostgreSQL)`
  - 理由: AIがSQLスキーマや連携コードを完璧に出力できる。トリガー機能を使えば「予約が登録されたら日割り売上を自動計算」などのロジックをDB側に持たせることも可能。
- **UIコンポーネント**: `Tailwind CSS` + `shadcn/ui`
  - 理由: AIに「shadcnを使って美しいカードとテーブルを作って」と指示するだけで、モダンなUIが完成する。
- **グラフ描画**: `Recharts`
  - 理由: Reactと相性が良く、AIが得意としている。

## 4. データベース構成（コアとなる3テーブル）

AIが混乱しないよう、最初は以下の3つのテーブルのみでMVP（最小限の機能）を構築します。

1. **`properties_inventory` (施設・部屋・在庫カレンダー)**
   - 用途: 部屋ごとのカレンダー表示用（空室状況、料金）
2. **`bookings` (予約情報マス)**
   - 用途: Beds24から取得した予約原本（チェックイン、アウト日、ゲスト名、総額）
3. **`daily_revenues` (日割り売上・稼働状況)** ※重要
   - 用途: 「予約情報」から日ごとに分解されたデータ。ダッシュボードのグラフ表示用。

> **💡 AIへの開発指示のコツ**
> 予約情報(`bookings`)が追加・更新されたら、Supabase側またはNext.jsのAPI側で自動的に日割り計算を行い、`daily_revenues` を更新するように指示します。これにより、フロントエンドは「単に `daily_revenues` のデータを出力するだけ」になり超高速に動作します。

## 5. API連携とデータ更新戦略（Beds24特化）

Beds24の「5分間で100クレジット」という厳しい制限とタイムゾーンを考慮し、以下のように同期を行います。

1. **基本は「Webhook」駆動（即時反映）**
   - Beds24側で設定。予約が入ったりキャンセルされた直後、Next.jsのAPI Router (`/api/webhooks/beds24`) がデータを受け取り、Supabaseのデータを更新する。これがメインの更新ルート。
2. **「1日1回」の深夜バッチ同期（ズレ補正＆在庫取得）**
   - Vercel Cron などを使用。在庫(inventory)状況や、Webhookで漏れた予約の差分を深夜に取得し、データを強制的に同期させる。
3. **フロントエンドからの直API叩きは厳禁**
   - ブラウザから Beds24 API は絶対叩かず、常に自社の Supabase からデータを読む（キャッシュ戦略）。

## 6. バイブコーディングの開発フェーズ（AIプロンプトの流れ）

以下の手順でAIに段階的にプロンプト（指示）を出していくと、エラーなく完成まで辿り着けます。

### フェーズ1: データベース設計と初期設定
*「Next.js App RouterとSupabaseを使用します。Beds24連携のための bookings, daily_revenues, properties_inventory の3つのテーブルのSQLを作成し、TypeScriptの型定義を生成してください。」*

### フェーズ2: Webhook受信用エンドポイントの作成
*「Next.jsの /api/webhooks/beds24 にPOSTエンドポイントを作成してください。Beds24からの予約更新データをパースし、Supabaseの bookings テーブルを更新。同時に、宿泊日数で総額を割って daily_revenues テーブルを日割りデータで更新する（または削除して再作成する）処理を書いてください。タイムゾーンはJST(date-fns等を利用)で厳密に処理してください。」*

### フェーズ3: 在庫同期バッチのエラー対応付き作成
*「Vercel Cronを用いた深夜の在庫取得バッチAPIを作ります。Beds24 API V2 (/inventory/rooms/calendar) を叩き、properties_inventory を更新します。Beds24のAPIレートリミット（429エラー）を考慮した指数バックオフによるリトライ処理を含めてください。」*

### フェーズ4: ダッシュボードUIの生成 (shadcn/ui活用)
*「shadcn/ui と Tailwind CSS, Recharts を使って、柴又の民泊用の美しい管理ダッシュボードを作成してください。Supabaseから当月の daily_revenues と bookings の直近一覧を取得し、『日別の売上グラフ』と『直近のチェックイン予定一覧』、『本日の部屋ごとの空室状況』を見やすく配置してください。」*
