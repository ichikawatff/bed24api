1. 開発目的

Beds24 の標準画面は多機能ですが、現場で見たい情報だけを横断して確認するにはやや重いです。
そのため本システムでは、Beds24 API v2 から必要情報を取得し、自社専用UIで以下を実現します。

施設別・日付別・予約別の見やすい可視化
予約状況、宿泊者情報、売上概況の即時把握
Beds24を直接触らなくても、日常確認業務の大半を自社画面で完結
将来的に 帳票出力・CSV連携・通知連携 へ拡張できる構成にする

Beds24 API v2 は bookings / inventory / properties / accounts を扱え、GET /bookings、GET /properties、GET /inventory/rooms/availability、GET /inventory/rooms/calendar 等が用意されています。

3. 前提条件・API前提
3.1 認証方式

Beds24 API v2 では、最初に Invite Code または Long Life Token を用意し、Invite Code を使う場合は GET /authentication/setup で token と refresh token を取得します。通常APIは token: {token} ヘッダで認証し、refresh token で GET /authentication/token を呼んで新しい token を再発行します。token は 24 時間で失効し、refresh token は 30 日以内に使われていれば継続利用できます。

3.2 スコープ

必要な scope は少なくとも以下です。

read:bookings
read:bookings-personal
read:inventory
read:properties
必要に応じて read:accounts

個人情報参照には bookings-personal、財務情報参照には bookings-financial が追加で必要です。scope は Invite Code 作成時に決まり、後から変更できません。

3.3 API利用制限

Beds24 API のクレジット制限は、既定で 5分あたり100 credits です。制限は token単位ではなく account単位 で共有され、レスポンスヘッダ x-five-min-limit-remaining、x-five-min-limit-resets-in、x-request-cost で状況確認できます。高頻度取得では polling を増やさず、webhook を使う設計が推奨されます。

3.4 webhook

Beds24 は新規予約やメッセージ等の高頻度取得について webhook 利用を推奨しています。Booking webhook は API v2 で利用でき、Beds24 側設定および場合により support 対応が必要です。なお changelog 上、2024-05-28 に booking webhook へ personal information が追加されています。

4.2 推奨技術スタック

既存のご希望と親和性が高いので、初版は以下を推奨します。

Frontend: Next.js
Backend: Next.js API Routes または Node.js API
DB: PostgreSQL
ORM: Prisma
Auth: NextAuth または管理者向け簡易認証
Hosting: Vercel + DB（Supabase / Neon / RDS 等）

理由は、一覧・詳細・検索・API中継・管理画面を一体で持ちやすく、将来の帳票や通知拡張もやりやすいからです。
ここは技術的には任意ですが、Beds24 側の制限を考えると、クライアントから直接 Beds24 を叩かず、必ず自社サーバー経由にしてください。これは token 管理と API 制御のため必須です。Beds24 の token は認証ヘッダとして使うため、ブラウザ直叩きは適しません。

13. 実装上の重要判断

この案件で重要なのは次の3点です。

13.1 Beds24直読みではなく、自社DBキャッシュ前提にする

Beds24 API は credit 制限があるので、毎画面ロードで直接叩く設計は弱いです。同期してDB表示が正解です。

13.2 webhookを主、定期同期を従にする

予約更新を即時反映したいなら polling より webhook が合理的です。Beds24 も高頻度取得で webhook を勧めています。

13.3 まずは「閲覧専用」に切る

初版で書き込みまでやると、scope、競合、誤更新、検証工数が一気に増えます。
最初は閲覧専用 + 同期安定化までに絞るのが堅いです。ここは仕様上の提案です。

15. 結論

この開発は十分現実的です。
むしろ Beds24 API v2 の仕様を見る限り、**「予約」「施設」「在庫」を自社画面で閲覧最適化する」**用途にはかなり合っています。
ただし、雑に作ると APIクレジット制限 で詰むので、自社DB保持 + webhook + 差分同期 を前提にした設計にするべきです。そこを外すと、あとで地味に痛いです。
